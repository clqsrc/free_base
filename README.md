# free_base
golang 我使用多年，这是个力求在标准版中尽量实现手动管理内存释放的库，并且尽量简单降低维护心智


理论上应该叫 free 不过，因为后面还会实现类似于 delphi 释放 form 下子控件的类似另外一个库，所以还是区分一下，奸库大概率会叫 free_form 。

其实两个库都已经实现，只是我的项目太多太乱，有些版本已经丢失 :) 所以还是放一个上来吧，省得老是重写。

让我下定决心正式实现它是因为 C# 的 CA2000（Dispose objects before losing scope） 我已经多次尝试，在 golnag 下是实现不了的。

有很多库和 ai 宣称可以实现，但很遗憾目前是不行的，如果哪天 go 语言界实现 CA2000 那这个项目基本上可以退役一半了。【现在唯一已知的就是据说 SonarLint 的收费版可以实现，不过我未证实】

目前最接近这一目标的应该是 tinygo 的字符串实现，很值得大家看一下它的实现。

所以折腾了这么多年，我觉得目前最好的方式还是简单点：回到最初的地方来，尽量不要让 golang 自动 gc 释放内存就可以了，幸好 golang 中是提供了足够的工具来实现这一目标的，这个项目基本上可以看作这些工具的小集合。

实际上对于我来说主要是用来管理 CGO 生成的 UI 控件，我自己在使用 govcl 上有惨重的经验教训，所以手动管理这些资源是刚需中的刚需。但是很显然 CGO 中问题的产生和其他场景是一样的，只是大多数情况被 GC 延时释放掩盖了，如果 gc 回收能实现实时回收，很多网友就不会觉得我的这个工作无关紧要了 

所以，是的，我就是要将 go 当做手动释放内存的语言来用 :) 随便你们怎么想 :)

--------------------------------------------------------

下面说下用法，其实用两个函数就足够了： func New_(obj interface{})  和 func Delete_(obj interface{}) 不过这里有一个坑，那就是 New_ 参数必须是指针。理论上应该可以从语法上进行限制，不过那到下个版本再说吧。

>Tips
>增加了 New_T/Delete_T 据说在 go 1.18 后可以应对上面的情况，不过还未大范围使用。

另外，给大家一个忠告，不要实现太复杂的复刻，相信我，当逻辑太过复杂时这些内存管理是非常不好控制的。一定要保持它的简单性！！！ 切记，切记 :)
